# 安全保障机制（Safeguards）

修复流程中的中等和轻度缺陷。

---

## 缺陷 #5 修复：source_hash 雪崩

### 问题

开发规划等文档是多个任务的 design_source，一次修改可能导致大量 Spec 标记为 stale。

### 解决方案：细粒度锚点 + 影响分级

**规则 1：Spec 的 design_source 必须使用锚点**

```yaml
# 错误：指向整个文件（任何修改都触发 stale）
design_source: 开发规划.md

# 正确：指向具体章节（只有该章节变化才触发）
design_source: 开发规划.md#day-2-3-噪声与高度图
```

**规则 2：source_hash 只计算锚点对应章节的内容**

**规则 3：stale 影响分级**

| 级别 | 判定条件 | 处理方式 |
|------|---------|---------|
| **无影响** | 修改了格式/措辞，未改接口/约束 | 更新 hash，Spec 内容不变 |
| **轻影响** | 修改了非接口部分（如实现建议） | 更新 hash + 审查 Spec 是否需要调整 |
| **重影响** | 修改了接口签名/边界约束 | 必须更新 Spec + 检查已有代码是否需要修改 |

## 缺陷 #6 修复：Never Again 规则质量

### 问题

Agent 自己写规则，可能太窄（只防个案）或太宽（误拦正常代码），且"只增不删"。

### 解决方案：规则生命周期管理

**规则 1：规则需要人工确认**

Agent 写完规则后，在评审报告中标注「新增规则」，人工验收时一并审核：
- 规则是否足够通用？
- 检查方法是否可靠？
- 是否会误拦正常代码？

**规则 2：规则可以修订（但不删除）**

```markdown
# RULE-001: GPU Buffer 必须在 finally/defer 中释放
## 修订历史
- v1 (2026-04-10): 初始版本
- v2 (2026-04-15): 放宽条件——RAII 包装类内部可免除 try-finally
```

**规则 3：规则可以标记为"休眠"**

如果规则频繁误报（3 次以上），不删除但标记休眠：

```markdown
status: dormant  # 休眠，不参与自动扫描
dormant_reason: "RAII 重构后不再适用原始检查模式"
dormant_date: 2026-04-20
```

休眠规则不参与 Step 0 扫描，但保留在库中供参考。

**规则 4：阶段性清理**

每个阶段结束时，人工 review 全部规则：
- 确认每条规则仍然有效
- 合并重复/相似规则
- 休眠已过时的规则

## 缺陷 #7 修复：并行 Agent 文件冲突

### 问题

多个 Agent 同时编码 ⚡ 并行任务时，可能同时修改 TASKS.md/STATUS.md。

### 解决方案：任务锁定协议

**规则 1：开始编码前锁定任务**

Agent 在 TASKS.md 中将任务状态改为 🟡 并标注 Agent ID：

```
| T-003 | 基础数学库 | 🟡 Agent-1 | ...
```

**规则 2：同一任务不可被两个 Agent 同时编码**

开始前检查目标任务是否已被锁定。

**规则 3：共享文件串行更新**

| 共享文件 | 规则 |
|---------|------|
| TASKS.md | 只在 Session 开始（锁定）和结束（更新状态）时修改 |
| STATUS.md | 只在 Session 结束时更新 |
| patterns.md | 只在 Session 结束时追加 |

**规则 4：推荐 Git Worktree 隔离**

并行 Agent 使用独立 worktree：

```bash
git worktree add ../terrain-agent-1 -b agent/T-003
git worktree add ../terrain-agent-2 -b agent/T-004
```

完成后合并到主分支，TASKS.md 串行更新。

## 缺陷 #8 修复：回归测试越来越慢

### 问题

"跑全量测试"在项目后期可能很慢，每次评审都全量跑效率低。

### 解决方案：分级测试策略

| 级别 | 内容 | 触发时机 | 时间预算 |
|------|------|---------|---------|
| **快速测试** | 本模块单元测试 + 直接依赖的接口测试 | 增量检查点（每次） | <30s |
| **标准测试** | 全部单元测试 | 评审门禁（Step 3） | <5min |
| **完整测试** | 全部单元 + 集成 + 性能基线 | 阶段结束 / 人工验收前 | 不限 |

**回归验证规则更新**：

- 增量检查点：跑快速测试
- 评审门禁 Step 3：跑标准测试（全部单元测试）
- 阶段结束：跑完整测试（含集成测试和性能基线）
- 任何级别的测试失败都必须修复

## 缺陷 #9 修复：性能基线冷启动

### 问题

首次基线没有参考，可能第一版就慢。

### 解决方案：绝对阈值 + 相对基线

**双重标准**：

| 标准 | 说明 |
|------|------|
| **绝对阈值** | 预先定义的性能上限，首次必须达标 |
| **相对基线** | 后续改动不劣化超过 10% |

**绝对阈值示例**（在 Spec 的验收断言中定义）：

```markdown
- PERF: 1024x1024 噪声生成 < 200ms (CPU) / < 50ms (GPU)
- PERF: 1024x1024 侵蚀 100 轮 < 2000ms (CPU) / < 200ms (GPU)
- PERF: 内存占用 < 输入尺寸 × 4 倍
```

首次基线录入时，如果未达到绝对阈值，评审不通过。

## 缺陷 #10 修复：缺少回滚机制

### 问题

done 任务后来发现方向错了，没有回滚流程。

### 解决方案：任务回退协议

**回退状态**：

```
🔵 done ──发现重大问题──→ 🔙 reverted ──重新规划──→ 🔴 planned（新任务 ID）
```

**回退流程**：

```
Step 1: 人工判定需要回退（Agent 不可自行决定）
Step 2: 在 TASKS.md 中标记原任务为 🔙 reverted，注明原因
Step 3: 创建新任务（新 ID），继承原任务的依赖关系
Step 4: 评估下游影响——依赖原任务 exports 的任务是否需要调整
Step 5: 原任务的代码保留在 git 历史中（不删除），但标记为废弃
Step 6: 写 Never Again 规则，记录为什么原方案不行
```

**下游影响评估**：

当一个已完成任务被回退时，必须检查：
- 哪些任务 imports 了它的 exports？
- 这些下游任务是否已经开始编码？
- 是否需要连带回退？

影响评估结果记录在 STATUS.md 的「已知问题」中。
