# Agent 编码完整流程

从任务开始到完成的端到端流程。

## 任务级别判定（首先执行）

Agent 开始任务时先判定级别，决定流程深度：

| 级别 | 条件 | 流程深度 |
|------|------|---------|
| **L1 轻量** | 预估 <3h，接口已定义，算法成熟 | 简化流程 |
| **L2 标准** | 预估 2-6h，有新接口或跨模块依赖 | 标准流程 |
| **L3 重度** | 预估 >6h，新算法/GPU/架构决策 | 完整流程 |

参见：[human-review-tiers.md](human-review-tiers.md)

## L3 完整流程（L1/L2 见下方简化版）

```
┌──────────────────────────────────────────────────────┐
│  Phase 0: 设计评审                                    │
│  Agent 输出设计方案 → 人工确认方向                     │
│  参见：spec-and-design.md 第一部分                     │
└──────────────────┬───────────────────────────────────┘
                   │ 方向确认
                   ▼
┌──────────────────────────────────────────────────────┐
│  Phase 1: Spec 生成                                   │
│  设计方案 → 编码契约（接口签名+边界约束+验收断言）     │
│  计算 source_hash，解析 exports/imports               │
│  人工确认 Spec → 任务状态 → spec_ready                │
│  参见：spec-and-design.md 第二部分                     │
└──────────────────┬───────────────────────────────────┘
                   │ Spec 确认
                   ▼
┌──────────────────────────────────────────────────────┐
│  Phase 2: 测试先行（TDD）                             │
│  按 Spec 验收断言编写 3-5 个测试 + 边界条件测试        │
│  测试能编译但全部 FAIL → 人工确认测试合理性            │
│  参见：test-first.md                                  │
└──────────────────┬───────────────────────────────────┘
                   │ 测试确认
                   ▼
┌──────────────────────────────────────────────────────┐
│  Phase 3: 编码（增量检查点 + 自检门）【编码 Agent】    │
│  查 patterns.md → 按模式编码                          │
│  查 rules/ → 加载相关维度的规则子集                    │
│  每完成一个核心函数 → 自检门(H/I/G) + 编译 + 快速测试  │
│  参见：coding-and-checkpoints.md                      │
└──────────────────┬───────────────────────────────────┘
                   │ 全部测试 PASS + 自检门通过
                   ▼
┌──────────────────────────────────────────────────────┐
│  Step 0: Never Again 规则扫描 【编码 Agent】           │
│  加载当前任务相关维度的规则子集                        │
│  任何规则未通过 → 直接打回修复                         │
│  参见：never-again.md                                 │
└──────────────────┬───────────────────────────────────┘
                   │ 全部通过
                   ▼
         ═══ 移交给评审 Agent（独立 Session）═══
         参见：cross-agent-review.md
                   │
                   ▼
┌──────────────────────────────────────────────────────┐
│  Step 1: 自动化预检 【评审 Agent】                     │
│  spec_verify → Spec ↔ 代码接口一致性                  │
│  auto_review → 可自动化评审项预检                     │
│  参见：automated-verification.md                      │
└──────────────────┬───────────────────────────────────┘
                   │ 全部通过
                   ▼
┌──────────────────────────────────────────────────────┐
│  Step 2: 13 项分级评审 【评审 Agent】                  │
│  Critical 5项必须10分 + Major 5项≥7分 + Standard 3项≥5│
│  举证式打分，每项附证据                               │
│  最多 5 轮迭代，第 3 轮触发根因分析                    │
│  参见：quality-gate.md                                │
└──────────────────┬───────────────────────────────────┘
                   │ 达标
                   ▼
┌──────────────────────────────────────────────────────┐
│  Step 3: 回归验证 【评审 Agent】                       │
│  跑标准测试（全部单元测试）                           │
│  任何测试失败 → 打回编码 Agent 修复                    │
│  参见：quality-gate.md 第六节                          │
└──────────────────┬───────────────────────────────────┘
                   │ 全部通过
                   ▼
┌──────────────────────────────────────────────────────┐
│  Step 4: 性能基线对比 【评审 Agent】                   │
│  绝对阈值（Spec PERF）+ 相对基线（劣化≤10%）          │
│  参见：quality-gate.md 第七节                          │
└──────────────────┬───────────────────────────────────┘
                   │ 达标
                   ▼
┌──────────────────────────────────────────────────────┐
│  Step 5: 人工验收                                     │
│  运行、视觉检查（视觉任务需截图集）、性能测量         │
│  随机抽检 2-3 项评审证据                              │
│  参见：human-review-tiers.md                          │
└──────────────────┬───────────────────────────────────┘
                   │ 通过
                   ▼
┌──────────────────────────────────────────────────────┐
│  Step 6: 收尾 【编码 Agent】                           │
│  新设计决策 → 沉淀到 patterns.md                      │
│  更新 STATUS.md / TASKS.md                            │
│  任务状态 → done                                      │
└──────────────────────────────────────────────────────┘
```

## L2 标准流程（省略设计评审）

```
Spec 生成 → 人工确认 Spec
    → 测试先行 → 编码+自检门
    → Never Again 扫描
    → 单 Agent 自评（Critical+Major 10项，举证式）
    → 回归验证 → 性能基线
    → 人工验收（Spec确认+最终验收）
    → 收尾
```

## L1 轻量流程（省略设计评审+Spec+测试确认）

```
编码+自检门 → Never Again 扫描
    → 单 Agent 自评（Critical 5项，举证式）
    → 快速回归测试
    → 人工验收（仅最终验收，可批量）
    → 收尾
```

L1 仍需编写测试（TDD），但跳过人工确认测试合理性步骤。

## 任务状态生命周期

```
🔴 planned ──Spec生成──→ 📋 spec_ready ──编码──→ 🟡 in_progress ──评审通过──→ 🔵 done
                                                                              │
                                                         发现重大问题 → 🔙 reverted
```

## 失败处理

| 场景 | 处理方式 |
|------|---------|
| 评审未达标 | 修复后从 Step 0 重新评审（可聚焦修复维度+关联维度） |
| 第 3 轮未达标 | 触发根因分析报告 |
| 第 5 轮未达标 | 红灯熔断，升级人工决策 |
| 人工验收不通过 | 编码 Agent 修复 + 写 Never Again 规则 + 重新走评审 |
| 设计源文档变更 | 检测 source_hash 漂移 → 更新 Spec → 必要时更新代码 |
| 已完成任务方向错误 | 走回退协议（safeguards.md） |

## Session 上下文传递

### 启动时必读

| 文件 | 作用 |
|------|------|
| **CLAUDE.md** | 规则、标准、项目架构 |
| **STATUS.md** | 当前进度、已完成模块、关键接口、已知问题 |
| **TASKS.md** | 任务注册表（状态、依赖、接口） |

### 结束时必须更新（仅编码 Agent）

| 文件 | 更新内容 |
|------|---------|
| **STATUS.md** | 进度、已完成模块、接口、已知问题、规则数、基线 |
| **TASKS.md** | 任务状态、产出接口、评审轮次、关联规则 |
| **patterns.md** | 新的设计决策沉淀（如有） |
