# 人工审核分级机制（Human Review Tiers）

防止人工成为流程瓶颈。按任务复杂度分级，不同级别需要不同程度的人工介入。

## 缺陷 #3 修复：人工瓶颈

### 三级审核制度

| 级别 | 适用任务 | 人工介入点 | 预计占比 |
|------|---------|-----------|---------|
| **L1 轻量** | 简单任务（<3h，接口明确，无新算法） | 仅最终验收 | ~40% |
| **L2 标准** | 中等任务（有新接口，有依赖） | Spec确认 + 最终验收 | ~40% |
| **L3 重度** | 复杂任务（新算法，跨模块，GPU） | 全流程人工介入 | ~20% |

### 各级别详细流程

#### L1 轻量（跳过设计评审和 Spec，跳过测试确认）

```
Agent 编码 → 自检门 → 评审门 → 回归 → 人工验收
```

条件：
- 预估 <3h
- 接口已在上游 Spec 中定义
- 算法成熟（如噪声函数是标准实现）

#### L2 标准（跳过设计评审，需要 Spec 确认）

```
Spec 生成 → 人工确认 Spec → 测试先行 → Agent 编码 → 全流程评审 → 人工验收
```

条件：
- 定义新接口
- 有跨模块依赖
- 预估 3-6h

#### L3 重度（全流程人工介入）

```
设计评审 → 人工确认 → Spec → 人工确认 → 测试 → 人工确认 → 编码 → 全流程评审 → 人工验收
```

条件：
- 新算法（侵蚀、贴合等）
- GPU Compute Shader
- 跨模块架构决策
- 预估 >6h

### 级别判定规则

Agent 在开始任务时自动判定级别，在评审报告开头声明：

```
审核级别：L2（标准）
理由：定义新接口 HeightmapGenerator，有上游依赖 T-007
```

人工可以在任何时候升级审核级别（但不能降级）。

### 批量确认

对于 L1 任务，人工可以批量验收：

- 累积 3-5 个 L1 任务后一次性验收
- 前提：每个任务都已独立通过评审门禁
- 验收时随机深入检查 1-2 个任务的评审证据
