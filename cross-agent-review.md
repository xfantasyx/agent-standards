# 交叉评审机制（Cross-Agent Review）

编码和评审由不同的 Agent 执行，天然消除自评偏差。

## 适用范围

| 任务级别 | 评审模式 | 说明 |
|----------|----------|------|
| **L3 重度任务** | **交叉评审（必须）** | 编码 Agent 与评审 Agent 分离，完整执行本文档流程 |
| **L2 中度任务** | 单 Agent 自评 + 人工抽检（默认） | 可选启用交叉评审，默认不启用 |
| **L1 轻量任务** | 单 Agent 自评 + 人工抽检（默认） | 可选启用交叉评审，默认不启用 |

### L1/L2 简化模式

L1/L2 任务默认采用以下简化流程：

1. **Spec 评审**（仅 L2，L1 无 Spec 不需要）：评审 Agent 审查 Spec 质量（必须）
2. **编码 Agent 自评**：编码完成后，同一 Agent 按分级评审标准自评（L1 评 Critical 5项，L2 评 Critical+Major 10项）
3. **人工效果验收**：人工运行程序看效果，不审查技术细节
4. **升级条件**：若自评发现严重问题（任意 Critical 维度 < 10 分），自动升级为交叉评审

以下内容仅针对 **L3 重度任务的交叉评审流程**。

## 核心原则

**写代码的 Agent 不评审自己的代码。**

```
编码 Agent（Session A）          评审 Agent（Session B）
┌────────────────────┐          ┌────────────────────┐
│ 设计评审            │          │                    │
│ Spec 生成           │          │                    │
│ 测试先行            │          │                    │
│ 编码 + 自检门       │          │                    │
│ Never Again 扫描    │          │                    │
│                    │          │                    │
│ ──── 编码完成 ────→ │  移交    │ 接收代码 + Spec     │
│                    │ ──────→ │ 执行 13 项评审       │
│                    │          │ 输出评审报告         │
│                    │  反馈    │                    │
│ 接收评审报告  ←──── │ ←────── │ ──── 评审完成 ────  │
│ 修复问题            │          │                    │
│ ──── 再次提交 ────→ │  移交    │ 重新评审            │
│                    │ ──────→ │                    │
└────────────────────┘          └────────────────────┘
```

## 职责划分

### 编码 Agent 负责

| 阶段 | 职责 |
|------|------|
| Phase 0 | 设计评审（输出设计方案） |
| Phase 1 | Spec 生成（编码契约） |
| Phase 2 | 测试先行（TDD） |
| Phase 3 | 编码 + 自检门 + 增量检查点 |
| Step 0 | Never Again 规则全量扫描 |
| 修复 | 根据评审报告修复问题 |
| 收尾 | 更新 STATUS.md / TASKS.md / patterns.md |

### 评审 Agent 负责

评审 Agent 在项目中承担两个角色：

**角色 1：Spec 评审（编码前）**

| 阶段 | 职责 |
|------|------|
| Phase 1 | 审查 Spec 质量（接口完整性、边界约束可测试性、imports/exports 一致性、验收断言覆盖度） |
| Phase 2（L3） | 确认测试用例合理性（覆盖度、边界条件、与 Spec 验收断言的映射） |

评审视角：**"如果我要按这个 Spec 编码，信息够不够？有没有矛盾？"**

**角色 2：代码评审（编码后）**

| 阶段 | 职责 |
|------|------|
| Step 1 | 自动化预检（spec_verify / auto_review） |
| Step 2 | 13 项分级评审（Critical全10 + Major≥7 + Standard≥5，举证式打分） |
| Step 3 | 回归验证（跑全量测试） |
| Step 4 | 性能基线对比 |
| 规则 | 评审中发现新错误模式 → 建议 Never Again 规则 |

**两个角色可以由同一个 Agent Session 连续执行**，先审 Spec，再审代码，积累项目上下文。L3 任务两个角色都必须执行；L1/L2 任务 Spec 评审必须执行，代码评审可选（默认单 Agent 自评）。

**重要：评审 Agent 不更新 STATUS.md / TASKS.md，这些文件只由编码 Agent 更新。** 评审 Agent 的职责仅限于评审和输出评审报告，不修改任何项目状态文件。

## 评审 Agent 启动上下文

评审 Agent 以独立 Session 启动，必须读取以下文件：

```
必读：
├── 被评审的代码文件（编码 Agent 产出）
├── 对应的 Spec 文件（specs/T-XXX.spec.md）
├── 对应的测试文件
├── quality-gate.md（13 项评审标准 + 举证要求 + 报告模板）
├── rules/（相关维度的 Never Again 规则）
└── patterns.md（项目模式库）

不读（避免受编码过程影响）：
├── 编码 Agent 的对话历史
└── 编码 Agent 的中间产物/调试记录
```

**关键：评审 Agent 不应该知道编码 Agent 遇到了什么困难、做了什么取舍。它只看最终产出和 Spec，以第三方视角评判。**

## 评审报告交接格式

评审 Agent 输出的报告包含两部分：

### Part 1：标准评审报告（分级评分）

按 quality-gate.md 第八节报告模板格式，每项附证据。

### Part 2：修复指令（如果未达标）

```
═══════════════════════════════════════════
  修复指令 — 交给编码 Agent
═══════════════════════════════════════════

修复项 1：
  维度：#X [维度名]
  问题：[具体描述]
  位置：[文件:行号]
  要求：[必须怎么改]

修复项 2：
  ...

建议新增规则：
  RULE-XXX: [规则描述]（编码 Agent 修复后编写）

═══════════════════════════════════════════
```

**Spec 问题反馈**：如果评审 Agent 发现 Spec 本身有问题（接口定义不合理、边界约束矛盾等），在修复指令中标记为「Spec 问题」。编码 Agent 收到后不直接修复代码，而是先修改 Spec（提交评审 Agent 重新审查），再根据新 Spec 修改代码。

编码 Agent 收到修复指令后：
1. 检查是否包含「Spec 问题」标记。如有，先修改 Spec 并提交评审 Agent 重新审查，待通过后再继续
2. 逐项修复代码问题
3. 编写建议的 Never Again 规则
4. 重新提交给评审 Agent

## 迭代规则

| 轮次 | 行为 |
|------|------|
| 第 1-2 轮 | 编码 Agent 修复 → 评审 Agent 重新评审 |
| 第 3 轮 | 若仍未达标，触发熔断根因分析（见 quality-gate.md 第五节） |
| 第 5 轮 | 红灯熔断，升级人工 |

## 修复后重新评审的范围

编码 Agent 修复问题并重新提交后，评审 Agent 按以下规则重新评审：

1. **从 Step 0 开始**：重新执行 Never Again 规则扫描，确保修复未引入已知错误模式
2. **聚焦评审**：评审 Agent **可以只重新评分被修复的维度 + 关联维度**，无需全部 13 项从零打分
3. **关联维度判定**：若修复涉及代码结构变更，需重新评审 #3 代码质量、#6 架构等关联维度；评审 Agent 自行判断关联范围并在报告中说明理由
4. **未修改维度**：之前已通过的维度，若代码未变动，可沿用上一轮评分

## 效率优化

### 评审 Agent 可复用

同一个评审 Agent Session 可以连续评审多个任务，积累项目上下文。

### 并行评审

当多个编码 Agent 并行工作时，可以：
- 一个评审 Agent 串行评审多个任务
- 或多个评审 Agent 各自负责不同任务

### 轻量任务简化

L1/L2 任务默认采用单 Agent 自评 + 人工抽检模式（见上方"适用范围"），不启用交叉评审。

## 实际操作方式

### 方式 A：手动开两个 Session

```
Terminal 1: claude（编码 Agent）
Terminal 2: claude（评审 Agent）
```

编码 Agent 完成后通知人工，人工在评审 Agent 中提交评审请求。

### 方式 B：自动化触发

编码 Agent 完成 Step 0 后，自动 spawn 一个评审子 Agent：

```
编码 Agent → Agent tool（subagent_type: review）→ 评审 Agent
          ← 评审报告 ←
```

### 方式 C：Skill 驱动

注册 `/review` Skill，编码 Agent 完成后调用，自动启动评审流程。
