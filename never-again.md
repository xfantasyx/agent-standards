# Never Again 规则引擎

每当 Agent 在评审或人工验收中犯错，**立即沉淀为一条永久规则**，并注册为可执行 Skill，确保同类错误在整个项目生命周期内不再发生。

## 规则总量上限

- **规则上限：100 条**。当已有规则数量达到 100 条时，新增规则必须附书面理由说明**为什么不能合并到已有规则中**，经人工确认后方可突破上限。
- 鼓励在新增前优先尝试合并到已有规则，保持规则库精简。

## 自动过期机制

- 每条规则记录**最后触发日期**（`last_triggered: YYYY-MM-DD`）。
- **6 个月未触发**的规则自动标记为 `dormant`（休眠），不参与自动扫描但保留在库中。
- 休眠规则如再次被人工/评审触发，自动恢复为 `active`。

## 规则分类与按需加载

- 每条规则必须按 13 项评审维度打标签（`tags: [#1, #5, #10]` 等）。
- Agent 执行评审或编码时，**只加载与当前任务相关的维度子集对应的规则**，而非全量加载。
- 标签对应关系参见 quality-gate.md 中的 #1～#13 维度定义。

## 规则写入时机

| 场景 | 写入者 | 生效条件 |
|------|--------|----------|
| 评审 Agent 打回（评审扣分） | 编码 Agent 在修复时编写规则 | 下一轮评审前，人工确认后生效 |
| 人工验收打回（人工反馈） | 编码 Agent 根据人工反馈编写规则 | 下一轮评审前，人工确认后生效 |

**关键：规则写完后不会立即生效，必须经过人工确认才纳入自动扫描。**

## 机制流程

```
Agent 犯错（评审扣分 / 人工反馈）
    │
    ▼
┌─────────────────────────────────────┐
│  Step 1：提炼规则                    │
│  从具体错误中抽象出通用规则          │
│  写入 rules/RULE-XXX.md             │
│  打上评审维度标签（#1～#13）          │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│  Step 1.5：人工确认                  │
│  人工审核规则的通用性和检查方法       │
│  确认后规则状态变为 active            │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│  Step 2：注册为 Skill               │
│  将规则转化为可自动执行的检查逻辑    │
│  注册到 Agent 的 Skill 列表中        │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│  Step 3：纳入评审门禁               │
│  后续评审按需加载相关维度的 Skill     │
│  确保 Never Again                   │
└─────────────────────────────────────┘
```

## 规则文件格式（rules/RULE-XXX.md）

```markdown
# RULE-XXX: [规则名称]

## 触发场景
[什么情况下会触发这个错误]

## 错误案例
[首次犯错的具体代码/行为]

## 正确做法
[应该怎么做]

## 检查方法
[自动化检查的具体步骤/命令/代码模式匹配]

## 元数据
- tags: [#X, #Y]（对应 13 项评审维度编号）
- status: active / dormant
- last_triggered: YYYY-MM-DD
- created: YYYY-MM-DD

## 来源
- Session: [哪个 Session 犯的错]
- 评审轮次: 第 N 轮
- 评审维度: #X [维度名称]
- 日期: YYYY-MM-DD
```

## Skill 注册规范

每条规则对应一个 Skill，Agent 在编码和评审时按需调用：

| 属性 | 说明 |
|------|------|
| **Skill 名称** | `check-rule-XXX`，与规则编号一致 |
| **触发时机** | 每次评审的第 0 步（在 13 项细则之前执行） |
| **执行方式** | 按当前任务涉及的评审维度，加载对应标签的规则子集进行扫描 |
| **失败处理** | 任何一条规则检查未通过，直接阻断评审，先修复再进入正式评审 |
| **累积效应** | 规则只增不删（可休眠），随着项目推进，Agent 的防错能力持续增强 |

## 运作规则

1. **即时沉淀**：犯错后当场写规则，不延迟、不批量处理
2. **抽象通用**：规则必须从具体错误中抽象出通用模式，而非仅针对个案
3. **可自动检查**：每条规则必须包含可执行的检查方法（grep 模式 / AST 扫描 / 测试用例）
4. **前置拦截**：规则检查在 13 项评审之前执行，不合格直接打回，不浪费评审时间
5. **人工确认**：Agent 写完规则后人工审核，确认通用性和检查方法的可靠性，**确认后才生效**
6. **人工可追加**：人工验收中发现的问题同样走此流程，由编码 Agent 编写规则并注册 Skill
7. **可修订不可删除**：规则可以修订（记录修订历史），但不能删除
8. **可休眠**：频繁误报（3 次以上）的规则标记为 dormant，不参与自动扫描但保留在库中
9. **自动过期休眠**：6 个月未触发的规则自动标记为 dormant
10. **阶段性清理**：每个阶段结束时人工 review 全部规则，合并重复、休眠过时规则
11. **总量控制**：规则总数上限 100 条，超出时必须附理由说明为何不能合并到已有规则
12. **按需加载**：Agent 只加载与当前任务相关维度的规则子集，避免全量扫描开销
13. **维度标签**：每条规则必须标注对应的评审维度（#1～#13），便于分类管理和按需加载
