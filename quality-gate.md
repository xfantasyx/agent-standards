# 质量评审门禁（Quality Gate）

每个 Agent Session 完成编码后，**必须执行质量评审**。评审采用 13 项维度打分（每项 0-10 分），按 **分级评分制** 判定通过与否。不再要求 130/130 满分，而是根据维度重要性设定不同及格线。

---

## 一、13 项评审维度与分级

### Critical（5 项）— 必须 10/10

关乎系统能否正确运行的底线维度，无妥协空间。

| # | 评审维度 | 满分标准（10/10） | 常见扣分点 | 必须提供的证据 |
|---|---------|-----------------|-----------|--------------|
| **1** | **编译与构建** | 零错误零警告通过编译，所有依赖正确解析 | 未处理的编译警告、缺失依赖、平台兼容问题 | 贴出编译命令和输出（零 warning） |
| **2** | **功能正确性** | 完整实现需求规格的所有功能点，边界条件正确处理 | 遗漏功能点、边界条件未覆盖、数值精度问题 | 列出每个需求点对应的代码位置 |
| **3** | **算法正确性** | 算法实现与理论/论文一致，数值稳定，无退化情况 | 公式实现偏差、数值溢出/下溢、特殊情况未处理 | 引用论文/参考，对比公式实现 |
| **5** | **内存安全** | 无泄漏、无越界、无悬空引用、资源生命周期明确 | 未释放 GPU 资源、数组越界、use-after-free | 列出所有资源创建/释放配对 |
| **13** | **Spec 一致性** | 接口签名/命名空间/导出符号与 Spec 完全一致 | 接口偏离 Spec 定义、类型签名与上游不匹配 | Spec 接口定义 vs 实际代码逐条对比 |

### Major（5 项）— 必须 ≥ 7/10

影响工程质量和可维护性的重要维度，允许合理 trade-off 但不可低于 7 分。

| # | 评审维度 | 满分标准（10/10） | 常见扣分点 | 必须提供的证据 |
|---|---------|-----------------|-----------|--------------|
| **4** | **性能与复杂度** | 时间/空间复杂度最优，无冗余计算，热路径已优化 | O(n²) 可优化为 O(n log n)、重复计算、不必要的内存分配 | 标注热路径的时间复杂度 |
| **6** | **错误处理** | 所有外部输入/IO/资源获取有错误处理，失败路径不崩溃 | 未检查文件打开、GPU buffer 分配失败未处理、除零未防护 | 列出所有外部调用的错误处理代码 |
| **7** | **接口设计** | API 语义清晰、参数命名准确、职责单一 | 函数过长（>50行）、参数过多、耦合其他模块内部状态 | 每个公共函数的参数数量和行数 |
| **9** | **模块解耦** | 模块间通过定义好的接口通信，无循环依赖，可独立编译测试 | 直接访问其他模块内部数据、全局可变状态、隐式依赖 | 列出模块间依赖方向 |
| **10** | **测试覆盖** | 关键路径有单元测试，边界条件有测试，测试可重复执行 | 无测试、测试依赖外部文件/网络、非确定性测试 | 测试用例数量 + Spec ASSERT 覆盖映射 |

### Standard（3 项）— 必须 ≥ 5/10

质量锦上添花的维度，允许更大弹性，但不可完全忽视。

| # | 评审维度 | 满分标准（10/10） | 常见扣分点 | 必须提供的证据 |
|---|---------|-----------------|-----------|--------------|
| **8** | **代码可读性** | 命名自解释、逻辑流清晰、复杂算法有注释引用论文/公式 | 魔法数字、单字母变量名、嵌套过深（>3层） | 声明无魔法数字、最大嵌套深度 |
| **11** | **GPU/并行安全** | 无数据竞争、同步正确、Compute Shader workgroup 尺寸合理 | Race condition、barrier 缺失、shared memory bank conflict | 列出所有 shared memory 和同步点 |
| **12** | **架构一致性** | 符合项目整体架构设计，遵循已有模式和约定 | 命名风格不统一、违反管线数据流方向、绕过既定接口 | 与已有模块的命名/模式对比 |

## 二、通过条件

| 条件 | 要求 |
|------|------|
| **Critical 全部满分** | 5 项 Critical 每项必须 10/10（合计 50/50） |
| **Major 全部达标** | 5 项 Major 每项必须 ≥ 7/10（合计 ≥ 35/50） |
| **Standard 全部达标** | 3 项 Standard 每项必须 ≥ 5/10（合计 ≥ 15/30） |

三个条件 **全部满足** 才算通过。任何一项不满足，必须修复后重新评审。

## 三、L1/L2/L3 分级评审范围

不同复杂度的任务适用不同评审级别：

| 评审级别 | 适用场景 | 评审范围 | 通过条件 |
|---------|---------|---------|---------|
| **L1 简化评审** | 小型修复、配置变更、文档修改 | 仅 Critical 5 项 | 5 项全部 10/10（50/50） |
| **L2 标准评审** | 常规功能开发、模块修改 | Critical + Major 共 10 项 | Critical 全 10 + Major 全 ≥ 7 |
| **L3 完整评审** | 新模块、核心算法、架构变更 | 全部 13 项 | Critical 全 10 + Major 全 ≥ 7 + Standard 全 ≥ 5 |

**注意**：L1 不评审 Major/Standard 维度，但这不代表可以违反这些维度的基本要求。L1 只是省略了对这些维度的正式评分，若人工效果验收发现问题仍可打回。

## 四、举证式评审（Anti-Bias）

Agent 自评存在天然的放水动机——打回意味着更多工作。为对抗偏差，每项评分 **必须附带具体证据**，无证据的得分视为 0 分。

### 举证要求

每项得分后必须提供上方表格中「必须提供的证据」列所列的内容。格式见下方报告模板。

### 自动化优先

可自动化检查的维度，必须用脚本验证而非自评：

| 阶段 | 自动化替代自评 |
|------|--------------|
| 编码开始时 | 编译检查（#1）自动化 |
| 第一轮评审后 | 将首次发现的可自动化问题立即写成检查脚本 |
| 阶段一结束 | #1/#5/#8/#10/#12/#13 应全部有脚本覆盖 |

### 人工效果抽检

人工验收时只看运行效果和截图，不抽检代码维度的评审证据（因为人工不懂技术）。技术质量由评审 Agent 闭环保障。

人工效果抽检内容：
- 运行程序看效果是否符合预期
- 视觉任务看截图是否自然
- 极端参数下程序是否崩溃

若人工发现效果不对：
1. 打回编码 Agent 修复
2. 编码 Agent 分析根因，写一条 Never Again 规则
3. 连续 2 次效果不达标 → 该类任务后续必须增加更多视觉/效果测试用例

## 五、熔断机制（Circuit Breaker）

防止评审陷入无限循环。最多 5 轮，超过则强制升级人工决策。

| 轮次 | 行为 |
|------|------|
| 第 1-3 轮 | 正常迭代：修复 → 重新评审 |
| 第 3 轮仍未通过 | **黄灯**：Agent 必须输出根因分析报告（见下方模板） |
| 第 4-5 轮 | 继续尝试，但每轮必须说明为什么之前修不好 |
| 第 5 轮仍未通过 | **红灯熔断**：停止迭代，升级到人工决策 |

### 根因分析报告（第 3 轮触发）

```
═══════════════════════════════════════════
  根因分析 — [Session 名称]
  已迭代 3 轮仍未通过
═══════════════════════════════════════════

持续不达标维度：#X [维度名]（当前得分 / 要求得分）

根因分析：
  □ 实现能力问题（算法太复杂，需要换方案）
  □ 设计缺陷（Spec 要求与实际矛盾）
  □ 依赖问题（上游接口不满足需求）
  □ 测试问题（测试用例本身有误）

建议：
  [具体的解决方向]

═══════════════════════════════════════════
```

### 红灯熔断后的人工决策

| 决策选项 | 适用场景 |
|----------|---------|
| **降级通过** | 该维度扣分合理但不影响功能，记录技术债，后续补 |
| **修改 Spec** | Spec 要求不合理，修改后重新编码 |
| **换方案** | 当前方案不可行，回到设计评审重新选型 |
| **拆分任务** | 任务太大/太复杂，拆成更小的子任务 |
| **人工介入** | 特定维度由人工辅助实现 |

### 技术债登记

降级通过的维度必须登记技术债：

```markdown
## 技术债 — TD-XXX

- 任务：T-XXX
- 维度：#X [维度名]
- 评审得分：X/10（降级通过，要求 Y/10）
- 原因：[为什么无法达标]
- 偿还计划：[何时、如何补齐]
- 登记日期：YYYY-MM-DD
```

技术债记录在 STATUS.md 的「已知问题」中，后续 Sprint 安排偿还。

## 六、回归验证

评审门禁包含回归测试，按分级策略执行：

| 级别 | 内容 | 触发时机 | 时间预算 |
|------|------|---------|---------|
| **快速测试** | 本模块单元测试 + 直接依赖的接口测试 | 增量检查点（每次） | <30s |
| **标准测试** | 全部单元测试 | 评审门禁 Step 3 | <5min |
| **完整测试** | 全部单元 + 集成 + 性能基线 | 阶段结束 / 人工验收前 | 不限 |

### 回归规则

| 规则 | 说明 |
|------|------|
| **零失败** | 任何级别的测试失败即判定不通过，必须修复 |
| **不可禁用** | 不允许通过注释/跳过/删除测试来"通过"回归验证 |
| **新增测试不豁免** | 本次 Session 新写的测试也纳入回归，确保自身一致 |
| **快速测试常跑** | 增量检查点每次都跑，成本低收益高。L1 任务仅跑快速测试（本模块单元测试 + 直接依赖的接口测试），时间预算 <30s。如果快速测试不存在（首个模块），跳过回归但必须在人工验收中说明 |
| **标准测试必跑** | 评审门禁阶段不可省略 |
| **完整测试兜底** | 阶段结束时跑完整测试，发现标准测试遗漏的问题 |

## 七、性能基线

关键模块首次通过评审时，记录性能基线。后续改动不允许无理由的性能劣化。

### 双重标准：参考阈值 + 相对基线

| 标准 | 说明 | 适用时机 |
|------|------|---------|
| **参考阈值** | 在 Spec 验收断言中预定义的性能上限（PERF: xxx < Nms），作为参考值而非硬性要求 | 首次基线录入时参考 |
| **相对基线（硬性）** | 后续改动不劣化超过 10% | 后续每次评审 |

### 基线规则

| 规则 | 说明 |
|------|------|
| **首次记录** | 关键模块首次通过评审时自动记录基线 |
| **参考阈值对照** | 首次基线录入时，如果未达到参考阈值，不直接判定失败，而是标记为"性能待优化"记入技术债 |
| **劣化阈值（硬性）** | 后续改动耗时或内存增长超过 10% 即判定不通过——这是唯一的硬性失败条件 |
| **合理劣化** | 因功能增加导致的合理劣化，需在评审报告中说明原因并更新基线 |
| **优化记录** | 性能优化后更新基线，记录优化幅度 |
| **阶段性评估** | 阶段结束时统一评估所有"性能待优化"项目，决定是否需要专项优化 |

### 基线文件格式

每个项目在根目录 `benchmarks/` 下存放基线文件：

```json
{
  "module": "module_name",
  "created": "YYYY-MM-DD",
  "updated": "YYYY-MM-DD",
  "metrics": {
    "time_ms": { "value": 45, "resolution": "1024x1024" },
    "memory_mb": { "value": 128, "resolution": "1024x1024" },
    "gpu_memory_mb": { "value": 64, "resolution": "1024x1024" }
  }
}
```

## 九、评审规则汇总

1. **分级达标**：Critical 必须满分，Major 必须 ≥ 7，Standard 必须 ≥ 5；不再要求全部满分
2. **举证必须**：每项评分必须附带证据摘要，无证据视为 0 分
3. **评审独立性**：评审时以批判视角审视代码，不因"已经花了很多时间"而降低标准
4. **迭代记录**：记录每轮评审的扣分项和修复内容，确保问题真正解决而非绕过
5. **熔断保护**：最多 5 轮，第 3 轮出根因分析，第 5 轮强制升级人工
6. **回归必跑**：评审门禁阶段标准测试必跑，不可省略
7. **性能守护**：有基线的模块，劣化超 10% 即不通过
8. **人工效果验收**：Agent 通过评审后，人工运行程序看效果验收；人工只看效果不审技术，技术质量由 Agent 闭环保障
