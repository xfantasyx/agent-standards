# 质量评审门禁（Quality Gate）

每个 Agent Session 完成编码后，**必须执行自动化质量评审**。评审采用 12 项工业级标准打分，每项 0-10 分，**总分 120 分，必须满分 120/120 才允许进入下一个任务**。任何一项未满分，必须迭代修复后重新评审，直到全部满分。

## 12 项评审细则

| # | 评审维度 | 满分标准（10/10） | 常见扣分点 |
|---|---------|-----------------|-----------|
| **1** | **编译与构建** | 零错误零警告通过编译，所有依赖正确解析 | 未处理的编译警告、缺失依赖、平台兼容问题 |
| **2** | **功能正确性** | 完整实现需求规格的所有功能点，边界条件正确处理 | 遗漏功能点、边界条件未覆盖、数值精度问题 |
| **3** | **算法正确性** | 算法实现与理论/论文一致，数值稳定，无退化情况 | 公式实现偏差、数值溢出/下溢、特殊情况未处理 |
| **4** | **性能与复杂度** | 时间/空间复杂度最优，无冗余计算，热路径已优化 | O(n²) 可优化为 O(n log n)、重复计算、不必要的内存分配 |
| **5** | **内存安全** | 无泄漏、无越界、无悬空引用、资源生命周期明确 | 未释放 GPU 资源、数组越界、use-after-free |
| **6** | **错误处理** | 所有外部输入/IO/资源获取有错误处理，失败路径不会导致崩溃 | 未检查文件打开、GPU buffer 分配失败未处理、除零未防护 |
| **7** | **接口设计** | API 语义清晰、参数命名准确、职责单一、调用方无需了解内部实现 | 函数过长（>50行）、参数过多、耦合其他模块内部状态 |
| **8** | **代码可读性** | 命名自解释、逻辑流清晰、复杂算法有必要注释引用论文/公式 | 魔法数字、单字母变量名、嵌套过深（>3层） |
| **9** | **模块解耦** | 模块间通过定义好的接口通信，无循环依赖，可独立编译测试 | 直接访问其他模块内部数据、全局可变状态、隐式依赖 |
| **10** | **测试覆盖** | 关键路径有单元测试，边界条件有测试，测试可重复执行 | 无测试、测试依赖外部文件/网络、非确定性测试 |
| **11** | **GPU/并行安全** | 无数据竞争、同步正确、Compute Shader workgroup 尺寸合理 | Race condition、barrier 缺失、shared memory bank conflict |
| **12** | **架构一致性** | 符合项目整体架构设计，遵循已有模式和约定，不引入架构偏移 | 命名风格不统一、违反管线数据流方向、绕过既定接口 |

## 评审规则

1. **零容忍**：任何一项低于 10 分即判定不通过，必须修复后重新全量评审
2. **不可跳过**：即使功能已正确运行，仍需逐项检查所有维度
3. **评审独立性**：评审时以批判视角审视代码，不因"已经花了很多时间"而降低标准
4. **迭代记录**：记录每轮评审的扣分项和修复内容，确保问题真正解决而非绕过
5. **人工验收前置**：Agent 满分通过后，人工仍需运行验收；评审门禁是最低标准，不是最终标准
