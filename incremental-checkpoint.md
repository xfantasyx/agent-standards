# 增量检查点（Incremental Checkpoint）

不要写完整个模块才检查。每完成一个核心函数就编译 + 跑测试，越早发现问题修复成本越低。

## 规则

```
编写函数 A → 编译 → 跑测试 → ✅ PASS
    │
    ▼
编写函数 B → 编译 → 跑测试 → ✅ PASS（A+B 都过）
    │
    ▼
编写函数 C → 编译 → 跑测试 → ❌ FAIL
    │
    ▼
立即修复 C → 重新跑测试 → ✅ PASS（A+B+C 都过）
    │
    ▼
继续下一个函数...
```

## 检查点触发条件

以下任一情况必须触发检查点（编译 + 跑已有测试）：

| 触发条件 | 说明 |
|----------|------|
| **完成一个公共函数** | 任何对外暴露的函数/方法写完后 |
| **完成一个算法核心** | 侵蚀步进、噪声生成、法线计算等核心逻辑 |
| **修改已有接口** | 改了函数签名、参数类型、返回值 |
| **编码满 30 分钟** | 无论进度如何，至少每 30 分钟编译一次 |

## 检查点失败处理

- **立即停下修复**，不继续编写新代码
- 修复后重跑全部已有测试，确认无回归
- 如果修复涉及改接口，检查所有调用方
