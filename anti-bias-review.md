# 对抗自评偏差机制（Anti-Bias Review）

Agent 自评存在天然的放水动机——打回意味着更多工作。本机制通过结构化约束对抗这个偏差。

## 缺陷 #1 修复：裁判员问题

### 对抗策略

#### 策略 A：举证式评审（核心机制）

Agent 不能直接打 10/10，必须为每一项提供**具体证据**。无证据的满分视为无效。

| 评审维度 | 必须提供的证据 |
|----------|--------------|
| #1 编译与构建 | 贴出编译命令和输出（零 warning） |
| #2 功能正确性 | 列出每个需求点对应的代码位置 |
| #3 算法正确性 | 引用论文/参考，对比公式实现 |
| #4 性能与复杂度 | 标注热路径的时间复杂度 |
| #5 内存安全 | 列出所有资源创建/释放配对 |
| #6 错误处理 | 列出所有外部调用的错误处理代码 |
| #7 接口设计 | 每个公共函数的参数数量和行数 |
| #8 代码可读性 | 声明无魔法数字、最大嵌套深度 |
| #9 模块解耦 | 列出模块间依赖方向 |
| #10 测试覆盖 | 测试用例数量 + Spec ASSERT 覆盖映射 |
| #11 GPU/并行安全 | 列出所有 shared memory 和同步点 |
| #12 架构一致性 | 与已有模块的命名/模式对比 |
| #13 Spec一致性 | Spec 接口定义 vs 实际代码 逐条对比 |

#### 策略 B：自动化优先

可自动化检查的维度，必须用脚本验证而非自评：

| 阶段 | 自动化替代自评 |
|------|--------------|
| 编码开始时 | 编译检查（#1）自动化 |
| 第一轮评审后 | 将首次发现的可自动化问题立即写成检查脚本 |
| 阶段一结束 | #1/#5/#8/#10/#12/#13 应全部有脚本覆盖 |

#### 策略 C：人工抽检

人工验收时不只看运行效果，**随机抽查 2-3 项评审维度的证据**，验证 Agent 是否真的达标。若发现放水：

1. 该维度直接判 0 分，打回重做
2. 写一条 Never Again 规则：`RULE-XXX: [维度名] 必须提供 [具体证据]`
3. 连续 2 次发现放水 → 该维度后续必须由人工打分

### 评审报告升级

评审报告中每项分数后必须附证据摘要：

```
 #  维度            得分    证据
─────────────────────────────────────────
 1  编译与构建       10/10  ✅ cmake --build: 0 errors, 0 warnings
 2  功能正确性       10/10  ✅ 5/5 需求点已实现（见清单）
 3  算法正确性       10/10  ✅ FBM 公式与 Perlin 1985 论文一致
...
```

无证据的行视为 0 分。
