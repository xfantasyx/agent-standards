# 设计评审与 Spec 规格层

本文档整合了编码前设计评审流程与 Spec 规格层规范，涵盖从设计方案到编码契约的完整前置流程。

---

## 第一部分：设计评审

复杂任务在编码前，Agent 必须先输出设计方案，经人工确认后再编码。避免方向错误导致整轮返工。

### 触发条件

以下任一条件满足即为"复杂任务"，必须先出设计方案：

- 预估编码时间 ≥ 3h
- 涉及新算法实现
- 涉及跨模块接口设计
- 涉及 GPU Compute Shader
- 涉及与外部格式/引擎的集成

简单任务（< 3h、接口明确、算法成熟）可跳过此步骤，直接编码。

### 设计方案格式

```
┌─────────────────────────────────────────┐
│ 设计方案 — [Session 名称]               │
├─────────────────────────────────────────┤
│ 目标：一句话说明要实现什么               │
│ 输入：接收什么数据/接口                  │
│ 输出：产出什么数据/接口                  │
│ 算法选型：用什么算法，为什么选它         │
│ 数据流：数据怎么流转（伪代码或图）       │
│ 接口定义：对外暴露的函数签名             │
│ 风险点：预判可能出问题的地方             │
│ 备选方案：如果主方案不行，Plan B 是什么  │
└─────────────────────────────────────────┘
```

### 流程

```
Agent 输出设计方案
    │
    ▼
人工确认方向（只看"要做的事对不对"，不审技术方案细节）
    │
    ▼ (方向确认后)
Agent 开始编码
```

---

## 第二部分：Spec 规格层

在设计评审和编码之间增加一个 **Spec 层**——将设计方案转化为机械可验证的编码契约。

### 核心理念

设计方案是人类可读的意图描述，Spec 是机器可验证的编码契约。Agent 按 Spec 逐条实现，评审时逐条对照。

```
设计方案（人类意图）
    │
    ▼ /spec-gen
┌─────────────────────────────────────┐
│  Spec 文件（编码契约）               │
│  ── 精确的接口签名                   │
│  ── MUST / MUST NOT 边界约束         │
│  ── 验收断言（可映射到测试用例）      │
│  ── source_hash 追踪设计变更         │
└──────────────┬──────────────────────┘
               │ /spec-code
               ▼
           编码实现
```

### 触发条件

以下任务必须先生成 Spec 再编码：

- 定义对外接口的模块
- 涉及跨模块依赖（有 imports）的任务
- 算法实现类任务
- 简单工具类/配置类任务可跳过

**L1 任务与 Spec 的关系**：L1 任务不需要生成 Spec（因为 L1 不定义新接口）。如果发现 L1 任务需要定义新接口，应先升级为 L2 再生成 Spec。

### Spec 文件格式

存放于项目 `specs/` 目录下，每个任务一个文件：

```markdown
---
task: T-008                              # 对应 TASKS.md 的任务 ID
title: "高度图生成器"
depends: [T-007, T-006]                  # 前置依赖
design_source: 开发规划.md#day-2-3       # 设计来源
source_hash: "fd91b244294a70db"          # 设计源内容 hash
output_files:                            # 必须产出的文件
  - src/heightmap/heightmap_generator.h
  - src/heightmap/heightmap_generator.cpp
exports:                                 # 本任务导出的接口
  - "HeightmapGenerator"
  - "HeightmapConfig"
imports:                                 # 消费的上游接口
  - "fbm()"                             # from T-007
  - "voronoi2d()"                        # from T-006
---

# 高度图生成器

## 概述
（2-3 句描述实现目标）

## 依赖类型（内联）
（从上游 Spec 提取的类型声明，标注真源）

## 接口定义
（完整的接口声明，可直接复制到源文件）

## 边界约束
- MUST: 支持 256x256 到 8192x8192 分辨率
- MUST: 输出值范围 [0.0, 1.0]，不出现 NaN/Inf
- MUST: 相同 seed + config 产生相同结果（确定性）
- MUST NOT: 在生成过程中分配堆内存超过输出 buffer 的 2 倍
- MUST: 支持多噪声层叠加，层数 ≥ 1

## 实现要求
（算法细节、性能要求、线程安全等）

## 文件结构
（目录布局 + 各文件职责）

## 验收断言
- ASSERT: 1024x1024 生成结果值域在 [0.0, 1.0] 内
- ASSERT: seed=42 两次生成结果逐像素相同
- ASSERT: 分辨率 256x256 生成耗时 < 100ms
- ASSERT: 8 层 FBM 叠加无数值溢出
- ASSERT: voronoi 权重为 0 时退化为纯 FBM
```

### source_hash 机制

追踪 Spec 与设计源文档之间的新鲜度：

source_hash 使用设计源文件的最后修改时间戳（ISO 8601 格式）。当 Spec 的 source_hash 与设计源文件的当前修改时间不匹配时，标记为 stale。

**用途**：
- 设计文档修改后，自动检测哪些 Spec 的 source_hash 不再匹配
- 不匹配的 Spec 标记为 `stale`，必须更新后才能继续编码
- 防止设计改了但代码还是按旧设计写的

### exports / imports 一致性验证

| 检查项 | 说明 |
|--------|------|
| **imports 可追溯** | 每个 imports 条目必须能追溯到依赖链中某个任务的 exports |
| **签名一致** | imports 的类型签名必须与上游 exports 逐字匹配 |
| **依赖完整** | 使用了上游接口但未声明 imports → 报错 |
| **无孤立 exports** | exports 未被任何下游 imports 引用 → 警告（可能是叶子模块） |

### Spec 自审查清单

Agent 写完 Spec 后必须逐条自查：
- [ ] 每个接口函数的参数类型、返回值、异常情况是否明确定义？
- [ ] 每条 MUST/MUST NOT 是否可测试（能写出对应的测试断言）？
- [ ] exports 列表是否完整（下游任务需要用的全部列出）？
- [ ] imports 是否与上游 Spec 的 exports 逐字匹配？
- [ ] 验收断言是否覆盖正常路径、边界条件、异常输入？
- [ ] 有没有遗漏的异常/错误处理路径？

### 边界约束维度表

每次写 Spec 时逐个维度检查是否需要约束（不是每个都需要，但每个都必须过一遍）：

| 维度 | 检查问题 |
|------|---------|
| 数值范围 | 输入/输出的值域是什么？ |
| 分辨率/尺寸 | 支持的最小/最大尺寸？ |
| 确定性 | 相同输入是否保证相同输出？ |
| 线程安全 | 是否支持并发调用？ |
| 资源限制 | 内存/GPU 显存上限？ |
| 空值/零值 | 输入为空/零时的行为？ |
| 性能预期 | 需要 PERF 参考阈值吗？ |
| 平台差异 | 有平台相关行为吗？ |
| 精度 | 浮点精度要求？ |

### Spec 独立评审（评审 Agent）

Spec 生成后，由独立的评审 Agent 审查（不是编码 Agent 自审）。评审视角：**"如果我要按这个 Spec 编码，信息够不够？有没有矛盾？"**

评审 Agent 使用与编码 Agent 相同的自审查清单和边界约束维度表进行审查，但以独立第三方视角执行。重点关注编码 Agent 可能遗漏的盲点：
1. 自审查清单 6 项是否全部达标（见上方）
2. 边界约束维度表 9 个维度是否逐项检查过（见上方）
3. 从"编码者视角"审查：如果我要按这个 Spec 编码，还缺什么信息？

**Spec 评审通过** → 任务状态 → spec_ready。

**Spec 评审不通过** → 评审 Agent 输出修改意见 → 编码 Agent 修改 Spec → 重新提交评审 Agent 审查。最多 3 轮迭代，超过则升级人工介入判断 Spec 方向是否需要调整。

### Spec 生成流程

```
Step 1: 选择任务（status 为 planned 或 spec_ready）
Step 2: 读取设计源文档，计算 source_hash
Step 3: 解析依赖链，提取上游 exports 类型声明
Step 4: 编写 Spec（接口定义 + 边界约束 + 验收断言）
Step 5: 自审查（Spec 自审查清单 + 类型一致性 + 接口签名匹配上游）
Step 6: 评审 Agent 审查 Spec（独立评审，非编码 Agent 自审）
Step 7: 更新任务状态 → spec_ready
```

### 任务状态扩展

引入 Spec 层后，任务状态从 3 态扩展为 4 态：

```
🔴 planned ──Spec生成──→ 📋 spec_ready ──编码──→ 🟡 in_progress ──评审通过──→ 🔵 done
```

| 状态 | 含义 | 允许的操作 |
|------|------|-----------|
| 🔴 planned | 已规划，尚未生成 Spec | 生成 Spec |
| 📋 spec_ready | Spec 已就绪，可开始编码 | 开始编码 |
| 🟡 in_progress | 编码进行中 | 编码、评审 |
| 🔵 done | 评审通过 + 人工验收 | 归档 |
