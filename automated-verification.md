# 自动化验证脚本规范

评审不能只靠 Agent 自评——需要独立的自动化脚本进行机械验证。每个项目应逐步建立自动化验证工具链。

## 验证脚本体系

```
tools/
├── validate.py              # 项目级确定性验证（跨文件一致性）
├── spec_verify.py           # Spec ↔ 代码接口一致性验证
├── auto_review.py           # 评审细则自动预检
├── acceptance_verify.py     # 验收断言 ↔ 测试用例覆盖率
└── dashboard.py             # 质量仪表盘生成
```

## 1. validate.py — 项目级确定性验证

跨文件一致性的机械检查，所有检查必须 100% PASS：

| 类别 | 检查项 | 说明 |
|------|--------|------|
| **任务一致性** | ID 唯一性 | TASKS.md 中无重复 ID |
| | DAG 无环 | 依赖关系无循环 |
| | 依赖引用有效 | depends 引用的任务 ID 存在 |
| **Spec 一致性** | source_hash 新鲜 | Spec 的 hash 与设计源文档匹配 |
| | Spec 覆盖 | spec_ready 及以后的任务必须有 Spec 文件 |
| | 前言完整 | Spec 必填字段齐全 |
| **接口一致性** | exports/imports 匹配 | imports 可追溯到依赖链 exports |
| | 签名一致 | 代码中实际接口与 Spec 声明一致 |
| **状态一致性** | 状态-产物匹配 | done 任务的 spec/代码/测试文件必须存在 |
| | 无孤立文件 | 代码文件必须被某个任务的 outputs 引用 |

## 2. spec_verify.py — Spec ↔ 代码一致性

交叉验证 Spec 的 exports 与实际代码中的声明：

- 扫描源文件提取 class/struct/function/type 声明
- 与 Spec exports 逐条比对
- 报告 [MATCH] / [MISSING] / [EXTRA]

## 3. auto_review.py — 评审自动预检

将 12 项评审细则中可自动化的部分脚本化：

| 可自动化 | 评审维度 | 检查方式 |
|----------|---------|---------|
| ✅ | #1 编译与构建 | 执行构建命令，检查 exit code 和 warning |
| ✅ | #5 内存安全 | 静态扫描资源创建/释放配对 |
| ✅ | #8 代码可读性 | 检查魔法数字、函数行数、嵌套深度 |
| ✅ | #9 模块解耦 | 检查 import 方向是否单向 |
| ✅ | #10 测试覆盖 | 检查 Spec 每条 ASSERT 有对应测试 |
| ✅ | #12 架构一致性 | 检查命名规范、文件组织 |
| ❌ | #2-4,6-7,11 | 需要语义理解，Agent 评审 |

## 4. acceptance_verify.py — 验收覆盖率

验证 Spec 中的验收断言是否被测试用例覆盖：

- 从 Spec 提取 `ASSERT:` 和 `PERF:` 行
- 从测试文件提取测试用例声明
- 模糊匹配（关键词重叠率 ≥ 0.65 判定覆盖）
- 输出覆盖率百分比，未覆盖的断言列出

## 5. dashboard.py — 质量仪表盘

生成项目质量概览报告：

| 面板 | 内容 |
|------|------|
| 进度概览 | 总任务数、按状态分布、完成率 |
| Spec 覆盖率 | 有 Spec 的任务 / 总任务 |
| 测试覆盖率 | 验收断言覆盖百分比 |
| Never Again 规则 | 规则总数、按维度分类 |
| 性能基线 | 各模块基线数据 + 趋势 |
| 依赖图 | 关键路径、可立即开始的任务 |

## 建设节奏

不需要一次全建好，按项目推进逐步建设：

| 阶段 | 建设内容 |
|------|---------|
| 编码开始时 | 构建脚本（编译 + 测试命令） |
| 第一个 Spec 完成后 | spec_verify.py |
| 第一轮评审后 | auto_review.py（从评审中发现的可自动化项开始） |
| 阶段一结束 | validate.py（跨任务一致性检查） |
| 阶段二开始 | acceptance_verify.py + dashboard.py |
