# 评审熔断机制（Circuit Breaker）

防止评审陷入无限循环。当迭代次数超过阈值时，强制升级到人工决策。

## 缺陷 #2 修复：无限评审循环

### 熔断规则

| 轮次 | 行为 |
|------|------|
| 第 1-3 轮 | 正常迭代：修复 → 重新评审 |
| 第 3 轮仍未满分 | **黄灯**：Agent 必须输出根因分析报告 |
| 第 4-5 轮 | 继续尝试，但每轮必须说明为什么之前修不好 |
| 第 5 轮仍未满分 | **红灯熔断**：停止迭代，升级到人工决策 |

### 根因分析报告（第 3 轮触发）

```
═══════════════════════════════════════════
  根因分析 — [Session 名称]
  已迭代 3 轮仍未满分
═══════════════════════════════════════════

持续不达标维度：#X [维度名]

根因分析：
  □ 实现能力问题（算法太复杂，需要换方案）
  □ 设计缺陷（Spec 要求与实际矛盾）
  □ 依赖问题（上游接口不满足需求）
  □ 测试问题（测试用例本身有误）

建议：
  [具体的解决方向]

═══════════════════════════════════════════
```

### 红灯熔断后的人工决策

| 决策选项 | 适用场景 |
|----------|---------|
| **降级通过** | 该维度扣分合理但不影响功能，记录技术债，后续补 |
| **修改 Spec** | Spec 要求不合理，修改后重新编码 |
| **换方案** | 当前方案不可行，回到设计评审重新选型 |
| **拆分任务** | 任务太大/太复杂，拆成更小的子任务 |
| **人工介入** | 特定维度由人工辅助实现 |

### 技术债登记

降级通过的维度必须登记技术债：

```markdown
## 技术债 — TD-XXX

- 任务：T-XXX
- 维度：#X [维度名]
- 评审得分：X/10（降级通过）
- 原因：[为什么无法满分]
- 偿还计划：[何时、如何补齐]
- 登记日期：YYYY-MM-DD
```

技术债记录在 STATUS.md 的「已知问题」中，后续 Sprint 安排偿还。
